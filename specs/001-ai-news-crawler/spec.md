## Clarifications

### Session 2026-03-22

- Q: 중요도 점수 산정 방식은? → A: 규칙 기반 (출처 신뢰도, 최신성, 키워드 빈도, 좋아요 수 등 가중치 합산). AI/ML 기반 분석은 사용하지 않음.
- Q: 크롤링 대상 소스 유형 및 수집 방식은? → A: HTML 스크래핑 방식. 대상은 AI 논문/기사 웹페이지 및 대형 IT 기업 기술 블로그. 한국어 콘텐츠 위주로 수집.
- Q: 애플리케이션 유형은? → A: 웹 애플리케이션 (브라우저 기반, 반응형 UI)
- Q: 게시물 중복 판별 기준은? → A: URL 우선 + 제목 유사도 보조. 원문 URL 일치 시 즉시 중복 처리, URL이 다르면 제목 유사도 검사로 추가 판별.
- Q: 예상 데이터 규모는? → A: 초기 크롤링 소스 10~20개, 일일 수집 게시물 100건 이하. 점진적 확장 가능.

## User Scenarios & Testing

### User Story 1 - AI 콘텐츠 탐색 및 중요도 기반 열람 (Priority: P1)

일반 사용자가 애플리케이션에 접속하면 AI 관련 기사, 논문, 기술 블로그 게시물 목록을 중요도 순으로 정렬하여 볼 수 있다. 사용자는 카테고리(기사/논문/블로그)별로 필터링하고, 제목이나 키워드로 검색할 수 있다. 각 게시물을 클릭하면 요약 정보와 원문 링크를 확인할 수 있다.

**Why this priority**: 크롤링된 콘텐츠를 중요도 순으로 보여주는 것이 이 애플리케이션의 핵심 가치이며, 다른 모든 기능(스크랩, 댓글, 좋아요)은 이 콘텐츠 목록 위에 구축된다.

**Independent Test**: 크롤링된 게시물 목록 페이지에 접속하여 중요도 순 정렬, 카테고리 필터, 검색 기능이 정상 동작하는지 확인하면 독립적으로 테스트 가능하다.

**Acceptance Scenarios**:

1. **Given** 크롤링된 게시물이 존재할 때, **When** 사용자가 메인 페이지에 접속하면, **Then** 게시물이 중요도 점수 기준 내림차순으로 표시된다
2. **Given** 다양한 카테고리의 게시물이 존재할 때, **When** 사용자가 '논문' 카테고리 필터를 선택하면, **Then** 논문 카테고리의 게시물만 표시된다
3. **Given** 게시물 목록이 표시된 상태에서, **When** 사용자가 키워드를 입력하여 검색하면, **Then** 제목이나 요약에 해당 키워드를 포함하는 게시물만 표시된다
4. **Given** 게시물 목록에서, **When** 사용자가 특정 게시물을 클릭하면, **Then** 게시물의 요약, 출처, 중요도 점수, 원문 링크가 표시된다

### User Story 2 - 관리자의 크롤링 소스 및 콘텐츠 관리 (Priority: P1)

관리자는 크롤링 대상 소스(뉴스 사이트, 논문 저장소, 기술 블로그)를 추가/수정/삭제할 수 있다. 또한 크롤링된 게시물의 중요도 점수를 수동으로 조정하거나, 부적절한 게시물을 삭제할 수 있다. 관리자는 일반 사용자를 관리(조회, 권한 변경, 비활성화)할 수 있다.

**Why this priority**: 크롤링 소스 관리 없이는 콘텐츠 수급이 불가능하고, 사용자 관리는 서비스 운영의 기본 요건이므로 P1으로 설정한다.

**Independent Test**: 관리자 계정으로 로그인 후 크롤링 소스를 추가하고, 해당 소스에서 게시물이 수집되는지, 사용자 목록을 조회하고 권한을 변경할 수 있는지 확인하면 독립적으로 테스트 가능하다.

**Acceptance Scenarios**:

1. **Given** 관리자로 로그인한 상태에서, **When** 새 크롤링 소스 URL과 카테고리를 등록하면, **Then** 해당 소스가 크롤링 대상 목록에 추가되고 다음 크롤링 주기에 수집이 시작된다
2. **Given** 크롤링된 게시물이 부적절할 때, **When** 관리자가 해당 게시물을 삭제하면, **Then** 일반 사용자에게 더 이상 표시되지 않는다
3. **Given** 관리자가 사용자 관리 페이지에 접속했을 때, **When** 특정 사용자의 역할을 변경하면, **Then** 해당 사용자의 권한이 즉시 반영된다

### User Story 3 - 게시물 스크랩 (Priority: P2)

일반 사용자가 관심 있는 게시물을 스크랩하여 나중에 다시 확인할 수 있다. 스크랩한 게시물은 별도의 '내 스크랩' 페이지에서 확인할 수 있으며, 스크랩을 해제할 수도 있다.

**Why this priority**: 콘텐츠를 저장하고 나중에 다시 볼 수 있는 기능은 사용자 리텐션에 중요하지만, 핵심 콘텐츠 열람 기능이 선행되어야 한다.

**Independent Test**: 로그인한 사용자가 게시물을 스크랩하고, '내 스크랩' 페이지에서 해당 게시물이 표시되는지 확인하면 독립적으로 테스트 가능하다.

**Acceptance Scenarios**:

1. **Given** 로그인한 사용자가 게시물 목록을 보고 있을 때, **When** 스크랩 버튼을 클릭하면, **Then** 해당 게시물이 스크랩 목록에 추가되고 스크랩 아이콘이 활성화 상태로 변경된다
2. **Given** 스크랩된 게시물이 있을 때, **When** 사용자가 '내 스크랩' 페이지에 접속하면, **Then** 스크랩한 게시물 목록이 스크랩 일시 기준 최신순으로 표시된다
3. **Given** 스크랩된 게시물이 있을 때, **When** 사용자가 스크랩 해제 버튼을 클릭하면, **Then** 해당 게시물이 스크랩 목록에서 제거된다

---

### User Story 4 - 게시물 좋아요 (Priority: P2)

일반 사용자가 게시물에 좋아요를 누를 수 있다. 좋아요 수는 게시물 목록과 상세 페이지에 표시되며, 이미 좋아요를 누른 게시물에 다시 누르면 좋아요가 취소된다.

**Why this priority**: 좋아요 기능은 사용자 참여도를 높이고, 커뮤니티 기반 중요도 판단에 활용될 수 있어 P2로 설정한다.

**Independent Test**: 로그인한 사용자가 게시물에 좋아요를 누르고, 좋아요 수가 증가하는지 확인하면 독립적으로 테스트 가능하다.

**Acceptance Scenarios**:

1. **Given** 로그인한 사용자가 게시물을 보고 있을 때, **When** 좋아요 버튼을 클릭하면, **Then** 좋아요 수가 1 증가하고 버튼이 활성화 상태로 표시된다
2. **Given** 이미 좋아요를 누른 게시물에서, **When** 좋아요 버튼을 다시 클릭하면, **Then** 좋아요가 취소되고 좋아요 수가 1 감소한다
3. **Given** 비로그인 사용자가 게시물을 보고 있을 때, **When** 좋아요 버튼을 클릭하면, **Then** 로그인 페이지로 안내된다

---

### User Story 5 - 게시물 댓글 (Priority: P3)

일반 사용자가 게시물에 댓글을 작성할 수 있다. 자신이 작성한 댓글을 수정하거나 삭제할 수 있으며, 관리자는 모든 댓글을 삭제할 수 있다.

**Why this priority**: 댓글은 사용자 간 소통을 가능하게 하지만, 콘텐츠 열람과 개인화 기능(스크랩, 좋아요)이 우선이므로 P3으로 설정한다.

**Independent Test**: 로그인한 사용자가 게시물 상세 페이지에서 댓글을 작성하고, 해당 댓글이 게시물 하단에 표시되는지 확인하면 독립적으로 테스트 가능하다.

**Acceptance Scenarios**:

1. **Given** 로그인한 사용자가 게시물 상세 페이지를 보고 있을 때, **When** 댓글을 작성하고 등록 버튼을 클릭하면, **Then** 댓글이 게시물 하단 댓글 목록에 표시된다
2. **Given** 자신이 작성한 댓글이 있을 때, **When** 수정 버튼을 클릭하고 내용을 변경하면, **Then** 댓글 내용이 업데이트되고 '수정됨' 표시가 나타난다
3. **Given** 자신이 작성한 댓글이 있을 때, **When** 삭제 버튼을 클릭하고 확인하면, **Then** 해당 댓글이 삭제된다
4. **Given** 관리자로 로그인한 상태에서, **When** 다른 사용자의 댓글을 삭제하면, **Then** 해당 댓글이 삭제된다

---

### User Story 6 - 사용자 회원가입 및 로그인 (Priority: P1)

사용자가 이메일과 비밀번호로 회원가입하고 로그인할 수 있다. 로그인하지 않은 사용자는 게시물 목록 열람만 가능하고, 스크랩/좋아요/댓글 기능은 로그인 후 사용할 수 있다.

**Why this priority**: 사용자 인증은 역할 기반 접근 제어, 스크랩, 좋아요, 댓글 등 모든 개인화 기능의 전제 조건이므로 P1으로 설정한다.

**Independent Test**: 회원가입 후 로그인하여 인증된 사용자 전용 기능(스크랩 버튼 등)이 활성화되는지 확인하면 독립적으로 테스트 가능하다.

**Acceptance Scenarios**:

1. **Given** 미가입 사용자가 회원가입 페이지에 접속했을 때, **When** 이메일과 비밀번호를 입력하고 가입하면, **Then** 계정이 생성되고 로그인 페이지로 이동한다
2. **Given** 가입된 사용자가 로그인 페이지에 접속했을 때, **When** 올바른 이메일과 비밀번호를 입력하면, **Then** 로그인되어 메인 페이지로 이동하고 사용자 이름이 표시된다
3. **Given** 비로그인 사용자가 게시물 목록을 보고 있을 때, **When** 스크랩/좋아요/댓글 기능을 사용하려 하면, **Then** 로그인이 필요하다는 안내와 함께 로그인 페이지로 이동한다

---

### Edge Cases

- 크롤링 대상 소스가 접속 불가능하거나 구조가 변경된 경우 어떻게 처리하는가?
- 중복된 게시물이 여러 소스에서 크롤링된 경우 어떻게 처리하는가?
- 사용자가 스크랩한 게시물이 관리자에 의해 삭제된 경우 스크랩 목록에서 어떻게 표시하는가?
- 크롤링된 게시물의 원문 링크가 더 이상 유효하지 않은 경우 어떻게 처리하는가?
- 한 사용자가 동일 게시물에 좋아요를 빠르게 반복 클릭하는 경우 어떻게 처리하는가?
- 관리자가 자신의 관리자 권한을 스스로 해제하려는 경우 어떻게 처리하는가?

## Requirements

### Functional Requirements

- **FR-001**: 시스템은 설정된 소스(AI 논문/기사 웹페이지, 대형 IT 기업 기술 블로그)에서 HTML 스크래핑 방식으로 한국어 콘텐츠를 위주로 주기적으로 크롤링해야 한다. 각 소스별 전용 파서를 등록하여 사용한다
- **FR-002**: 시스템은 크롤링된 게시물에 규칙 기반으로 중요도 점수를 자동 부여해야 한다. 점수 산정 요소: 출처 신뢰도, 최신성, 키워드 빈도, 좋아요 수 등의 가중치 합산 (AI/ML 미사용). 관리자는 가중치를 조정할 수 있다
- **FR-003**: 시스템은 게시물을 카테고리(기사/논문/블로그)별로 분류하고 필터링 기능을 제공해야 한다
- **FR-004**: 시스템은 게시물 제목 및 요약 내용을 기반으로 검색 기능을 제공해야 한다
- **FR-005**: 시스템은 이메일/비밀번호 기반 회원가입 및 로그인 기능을 제공해야 한다
- **FR-006**: 시스템은 관리자와 일반 사용자 두 가지 역할을 지원하고, 역할에 따라 접근 권한을 제어해야 한다
- **FR-007**: 관리자는 크롤링 소스를 추가, 수정, 삭제할 수 있어야 한다
- **FR-008**: 관리자는 크롤링된 게시물의 중요도 점수를 수동으로 조정하거나 게시물을 삭제할 수 있어야 한다
- **FR-009**: 관리자는 사용자 목록을 조회하고 역할 변경 및 비활성화를 수행할 수 있어야 한다
- **FR-010**: 로그인한 사용자는 게시물을 스크랩하고 스크랩 해제할 수 있어야 한다
- **FR-011**: 로그인한 사용자는 스크랩한 게시물을 별도의 페이지에서 확인할 수 있어야 한다
- **FR-012**: 로그인한 사용자는 게시물에 좋아요를 누르거나 취소할 수 있어야 한다
- **FR-013**: 시스템은 각 게시물의 좋아요 수를 표시해야 한다
- **FR-014**: 로그인한 사용자는 게시물에 댓글을 작성, 수정, 삭제할 수 있어야 한다
- **FR-015**: 관리자는 모든 사용자의 댓글을 삭제할 수 있어야 한다
- **FR-016**: 비로그인 사용자는 게시물 목록 열람 및 검색만 가능하고, 스크랩/좋아요/댓글 기능은 사용할 수 없어야 한다
- **FR-017**: 시스템은 여러 소스에서 수집된 동일 게시물을 자동으로 감지하고 중복을 제거해야 한다. 판별 기준: (1) 원문 URL 일치 시 즉시 중복 처리, (2) URL이 다른 경우 제목 유사도 검사로 추가 판별
- **FR-018**: 시스템은 크롤링 실패 시 관리자에게 알림을 제공하고, 실패한 소스에 대해 재시도해야 한다

### Key Entities

- **사용자(User)**: 이메일, 비밀번호, 닉네임, 역할(관리자/일반), 가입일, 활성 상태. 게시물에 대한 스크랩, 좋아요, 댓글 활동의 주체
- **게시물(Post)**: 제목, 요약, 원문 URL, 출처, 카테고리(기사/논문/블로그), 중요도 점수, 크롤링 일시, 좋아요 수. 크롤링 소스에서 수집된 AI 관련 콘텐츠
- **크롤링 소스(CrawlSource)**: URL, 이름, 카테고리, 활성 상태, 마지막 크롤링 일시, 크롤링 주기, 파서 식별자. HTML 스크래핑 대상 사이트 정보로, 소스별 전용 파서를 통해 콘텐츠를 추출한다
- **스크랩(Scrap)**: 사용자와 게시물의 관계. 사용자가 저장한 게시물 목록을 관리
- **좋아요(Like)**: 사용자와 게시물의 관계. 사용자 당 게시물 당 하나만 존재 가능
- **댓글(Comment)**: 작성자, 게시물, 내용, 작성일, 수정일. 게시물에 대한 사용자의 의견

## Success Criteria

### Measurable Outcomes

- **SC-001**: 사용자가 메인 페이지에 접속하여 중요도 순으로 정렬된 게시물 목록을 3초 이내에 확인할 수 있다
- **SC-002**: 관리자가 새 크롤링 소스를 등록한 후 다음 크롤링 주기 내에 해당 소스의 게시물이 수집된다
- **SC-003**: 사용자가 스크랩, 좋아요, 댓글 작성을 각각 2회 클릭 이내에 완료할 수 있다
- **SC-004**: 크롤링 시스템이 하루 최소 1회 이상 등록된 모든 소스에서 새 게시물을 수집한다
- **SC-005**: 중복 게시물이 사용자에게 표시되는 비율이 전체 게시물의 5% 미만이다
- **SC-006**: 회원가입부터 로그인 완료까지 2분 이내에 완료할 수 있다
- **SC-007**: 검색 결과가 1초 이내에 표시된다

## Assumptions

- 중요도 점수는 규칙 기반 가중치 합산으로 산정한다. 입력 요소: 출처 신뢰도, 게시물 최신성, 키워드 빈도, 사용자 참여도(좋아요 수). 관리자가 각 요소의 가중치를 조정할 수 있으며, 구체적인 가중치 기본값은 구현 단계에서 설계한다
- 크롤링 주기는 소스별로 관리자가 설정할 수 있으며, 기본값은 하루 1회로 가정한다
- 비밀번호는 업계 표준 보안 수준으로 저장 및 관리된다 (해싱, 솔팅 등)
- 이메일 인증은 초기 버전에서는 생략하고, 추후 추가하는 것으로 가정한다
- 관리자 계정은 초기에 시스템에서 직접 생성하며, 일반 사용자의 관리자 승격은 기존 관리자가 수행한다
- 게시물 목록은 페이지네이션을 지원하며, 기본 페이지 크기는 20개로 가정한다
- 초기 크롤링 소스는 10~20개로 시작하며, 일일 수집 게시물은 100건 이하로 예상한다. 이후 필요에 따라 점진적으로 확장한다
